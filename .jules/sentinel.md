## 2025-02-24 - [Path Traversal in Data Source Loading]
**Vulnerability:** A path traversal vulnerability existed in `src/data.rs` within the `resolve_source_path` function. The code directly joined the user-provided data source path (from the YAML/TOML configuration) with the base directory without sanitization. If an attacker provided a config file with a path containing `../` or an absolute path (e.g., `/etc/passwd`), the application would read and potentially leak the contents of arbitrary files from the host system during the load test.
**Learning:** The initial implementation assumed that the configuration file's author and the system running the load test were the same trusted entity. However, load testing tools are often run in CI/CD pipelines or as a service where users might upload their own configurations. Relying solely on `Path::join` without canonicalization or bounding checks exposes the system to path traversal attacks.
**Prevention:** Always normalize and canonicalize user-provided file paths before reading them. Use `std::fs::canonicalize` on both the base directory and the resolved path, and verify that the canonical resolved path strictly starts with the canonical base directory using `starts_with`. Furthermore, explicitly reject absolute paths if the design dictates that data sources must be relative to the configuration file's base directory.
